V11 2025-12-29 V0.01
{
- 修改了loaddata();函数的行为:
	- 使用mmap unmap方式映射保持1:1内存比例,当文件过大会采用流式加载避免峰值内存溢出
	- 内存比例请参考如下公式:	
	（<= MMAP_THRESHOLD = 256 KB，当前实现走映射分支）：峰值内存约为映射视图 + 目标字符串 ≤ 2 × file_size（再加上少量 std::string/堆元数据），所以膨胀比约 ≤ 2.0。
	例：61.3 KB → 峰值约 61.3KB (mapped) + 61.3KB (string) ≈ 122.6 KB，膨胀比 ≈ 2.0（实际物理占用可能更小，映射页按需载入）。
	大文件（> MMAP_THRESHOLD，当前实现走流式按块处理）：峰值内存 ≈ file_size + CHUNK + 少量开销，膨胀比约 ≈ 1 + CHUNK / file_size。CHUNK 当前值为 1 MB。
	公式：膨胀比 ≈ 1 + min(CHUNK, file_size) / file_size
	举例：
	file_size = 1 MB → ratio ≈ 1 + 1/1 = 2.0
	file_size = 10 MB → ratio ≈ 1 + 1/10 = 1.10（≈110%）
	file_size = 100 MB → ratio ≈ 1 + 1/100 = 1.01（≈101%）
	file_size = 1 GB → ratio ≈ 1 + 1/1024 ≈ 1.00098（≈100.1%）
	映射路径的“虚拟地址占用”不等同于即时物理内存占用：物理页面按访问载入，拷贝期间会短暂同时占用映射页与目标字符串的物理页面。实际工作集可能略高于上述估算。
	当前策略在小文件上保持低延迟但内存膨胀仍接近 2x；对中/大文件收益显著（随文件变大，膨胀比趋近 1）。
- 修改了trigger_async_save_two函数为后台写入的主线路,当任务累计到一定程度会触发备用线路。
- 优化了trigger_async_save的行为,将不再允许手动触发,被触发时将会开辟新的线路进行写入任务为备用线路,性能提升,采用了C++20新特性增强了异步写入的稳定性和效率。
- 增加了public变量tasks_Max当主异步线值大于等于此值时将会把主线路的后台任务提交给备用线路异步写入,默认值为1035可直接根据项目大小调整此值,以提升异步写入的效率和稳定性。
- 优化了异步写入的内存使用,大幅降低了异步写入时的内存峰值占用,因双线路保证,避免了死锁问题,当主线任务小于tasks_Max则恢复主线异步线路。
- 使用了完全原子写入覆写,每次写入都是整个内存,不存在需要管理线程写入混乱。

        # C++: const char* innerArrayStr = AList_Go(arrayStr, i);
        inner_ptr = AList_Go(array_str.encode("utf-8"), i)
        if not inner_ptr:
            continue

        inner_array_str = inner_ptr.decode("utf-8")

        # 内层：为每个内层数组分配一个临时 ID，并用 AList_fetch + AList_Get
        inner_id = f"inner_{i}".encode("utf-8")
        ALists(inner_id)  # 确保存在
        AList_fetch(inner_id, inner_array_str.encode("utf-8"))

        inner_size = AList_Size(inner_array_str.encode("utf-8"))
        print(f"inner[{i}] size =", inner_size)

        if inner_size < 6:
            continue

        # 像 C++ 那样逐个字段取出
        name_ptr  = AList_Get(inner_id, 0)
        total_ptr = AList_Get(inner_id, 1)
        price_ptr = AList_Get(inner_id, 2)
        per_box_ptr = AList_Get(inner_id, 3)
        day_ptr   = AList_Get(inner_id, 4)
        times_ptr = AList_Get(inner_id, 5)

        # 统一做一次解码与转换
        name      = name_ptr.decode("utf-8") if name_ptr else ""
        total     = float(total_ptr.decode("utf-8")) if total_ptr else 0.0
        price     = float(price_ptr.decode("utf-8")) if price_ptr else 0.0
        per_box   = float(per_box_ptr.decode("utf-8")) if per_box_ptr else 0.0
        day       = float(day_ptr.decode("utf-8")) if day_ptr else 0.0
        times     = float(times_ptr.decode("utf-8")) if times_ptr else 0.0

        medicines.append({
            "name": name,
            "total": total,
            "price": price,
            "perBox": per_box,
            "day": day,
            "times": times
        })
	```

V12 2026-5-1 V5.25
{
- 版本号 V12.05.25
- 将缓存管理器重构，从map更新为unordered_map，维护了FIFO的同时提升了安全边界和性能，尤其是在大数据存储场景下，缓存的效率提升显著。
- 缓存管理时将脏key缓存完全清除，并加入预定义功能。
- 预定义功能允许开发者预先定义一些常用的key，这些key将被优先保存在缓存中，确保在频繁访问时能够获得最佳性能。
	需注意，预定义一定要保证key是正确的否则会导致极大的BUG，如getvalue永远读取定义错误的数据。请保证value的正确性。
	用法分别有Predefine("title","header","key","value");和Predefine({"title","header","key","value"});
- 预定义功能特别适用于那些在项目中频繁访问的key但不会变化的配置。
- Predefine 适合：协议常量、版本号、固定配置、不变参数。不适合：每次请求都变的数据。
	最网络中也有着较大的用处如典型的用法是:
	```
	Predefine("Network", "Server", "IP", "");
	后续用 changevalue 填充真实 IP	
	之后 getvalue 就直接命中 Predefine 预留的缓存位置
	先占位、后更新、稳定命中，且永远不进入淘汰流程。
	并且预定义有模糊功能，先随便定义一个value，因为缓存管理只知道返回数据。因此value无论是多少，性能差距都并不大。
	而changevalue刚好是更新value的，此时再次更新，数据完全正确。在无需存档的情况下用处极强。尤其是在AAIC++和网络协议中。
	如先定义一个无意义的缓存
	as.Predefine("title", "header", "key", "value");
	修改缓存为正确的值
	as.changeValue("title", "header", "key", "127.0.0.1");
	之后就可以直接使用缓存了
	as.getvalue("title", "header", "key");
	加之预定义功能只会现到初始化中，因此在项目启动时就可以预定义好所有的常用key，确保在后续的使用中能够获得最佳性能和稳定性。
	```
